### Это тестовый интернет магазин, где я тестирую различные инструменты разработки

1. Функциональные требования (FR)
   1.1. Профили и роли (RBAC)
   Регистрация/Вход: Поддержка OAuth 2.0 (Яндекс ID, Google) и стандартной пары email/пароль.
   Баланс профиля:
   Покупатель: Просмотр остатка, история транзакций (списания за заказы, возвраты).
   Продавец: Просмотр доступных к выводу средств, история зачислений от продаж.
   Блокировка (Админ): При блокировке сессия пользователя должна мгновенно обрываться (revoke token). Заблокированный
   продавец не может редактировать товары, а его товары скрываются из каталога.
   1.2. Работа с товарами (Каталог и Инвентарь)
   Жизненный цикл товара (Продавец): Создание (черновик/опубликован), редактирование цен, остатков, описание (
   Markdown/HTML), загрузка до 5 фото.
   Каталог:
   Главная страница: Блоки «Скидки дня» (автоматически по наличию скидки > 0) и «Категории».
   Поиск и фильтры: По цене (min/max), категории, рейтингу, наличию скидки.
   Гость: Видит цену и описание, но при попытке «Добавить в корзину» получает редирект на /login.
   1.3. Заказы и Оплата
   Корзина: Сохранение состояния в БД (для авторизованных) или LocalStorage (для гостей, с мерджем при логине).
   Промокоды: Продавец создает промокод (процентный или фиксированный), ограничивает его по времени и количеству
   активаций.
   Имитация доставки: Интеграция с внешним API (например, JSONPlaceholder или тестовый контур логистической службы).
   Обновление статусов: Сформирован -> В пути -> Доставлен.
   1.4. Коммуникации
   Рассылки: Продавец рассылает уведомления только своим подписчикам (тем, кто покупал или добавил в «Избранное»).
   Админ — всем пользователям.
   Уведомления: Поддержка Push (через Firebase) или In-app уведомлений.
2. Нефункциональные требования (NFR)
   Для нагрузки 1000+ RPS одних слов мало, нужны технические метрики:
   Производительность:
   Время отклика API (P95) не должно превышать 150 мс для Read-запросов.
   Использование кэширования (Redis) для списка категорий и популярных товаров.
   Масштабируемость: Stateless архитектура бэкенда для горизонтального масштабирования в Kubernetes/Docker Swarm.
   Безопасность:
   Все пароли хешируются (Argon2/BCrypt).
   Использование HTTPS/TLS.
   Валидация всех входящих данных (защита от XSS/SQL Injection) на уровне API.
   Надежность: Хранение логов (ELK/Loki) и мониторинг (Prometheus/Grafana) для отслеживания ошибок в реальном времени.
3. Модерация товаров (Workflow продавца)
   В реальном маркетплейсе нельзя давать продавцу публиковать что угодно (например, запрещенные товары).
   Статусы модерации: При создании или значительном изменении цены/описания товар переходит в статус Pending (Ожидает
   проверки).
   Действие Админа: Админ видит очередь товаров на модерацию. Он может либо Approve (товар появляется в каталоге), либо
   Reject с указанием причины (причина улетает уведомлением продавцу).
   Авто-модерация: Если товар уже был проверен и продавец изменил только количество на складе, статус Active не должен
   слетать.
4. Процесс возврата товара (Refund & Balance)
   Самый сложный блок в плане логики транзакций.
   Инициация: Покупатель может нажать кнопку «Вернуть товар» в течение 14 дней, если статус доставки — Delivered.
   Финансовая логика:
   Средства на балансе продавца «замораживаются» (Hold) в момент покупки.
   При подтверждении возврата сумма списывается с баланса продавца (или из его будущего дохода) и возвращается на баланс
   покупателя.
   Промокод, использованный при покупке, не восстанавливается (бизнес-решение).
   Логистика: Статус заказа меняется на Returning -> Returned.
5. Лимиты и ограничения (Safety First)
   Защита системы от ошибок и злоупотреблений.
   Для Покупателя:
   Максимальная сумма одного заказа (например, до 500 000 руб.).
   Максимальное количество единиц одного товара в корзине (защита от перекупов).
   Для Продавца:
   Лимит на вывод средств (например, не чаще одного раза в сутки и не менее 1000 руб.).
   Максимальный вес/габариты товара (если сервис доставки имеет ограничения).
   Анти-фрод: Ограничение на количество попыток ввода промокода (не более 5 неверных вводов в час).

   Товары и категории
   Метод	Эндпоинт	Описание	Пример ответа
   GET	/api/products	Получить список товаров (с фильтрацией, пагинацией)	[{ "id": 1, "name": "...", "price": 2990, "category": "Мужская одежда", "description": "...", "image": "https://...", "inStock": true, "sellerId": 101, "status": "approved" }]
   GET	/api/products/{id}	Получить один товар по ID	как выше, но один объект
   GET	/api/categories	Список всех категорий	["Мужская одежда", "Женская одежда", "Обувь", "Аксессуары"]
   Параметры фильтрации для /api/products (через query string):

category – фильтр по названию категории

minPrice, maxPrice – фильтр по цене

search – поиск по названию/описанию

page, size – пагинация (по умолчанию page=0, size=20)

7. Загрузка изображений товаров
   Для формы добавления/редактирования товара нужна загрузка фото. Рекомендуется отдельный эндпоинт:

Метод	Эндпоинт	Описание
POST	/api/upload/image	Загрузить изображение (multipart/form-data). Ответ: { "imageUrl": "https://..." }
8. Форматы ошибок
   Все ошибки должны возвращать HTTP статус (400, 401, 403, 404, 500) и тело:

json
{
"error": "Краткое описание",
"message": "Детали для пользователя",
"timestamp": "2025-01-01T00:00:00Z"
}
9. Аутентификация
   Рекомендуемый способ: JWT токен, который фронт хранит в localStorage (или HttpOnly cookie). При каждом запросе (кроме /auth/login, /auth/register, публичных) токен должен передаваться в заголовке Authorization: Bearer <token>.

Если используете сессии на бэкенде (например, Spring Session), то кука сама подставится браузером, но для кросс-доменных запросов надо настроить CORS.
